こんにちは。PIVOTのテックリード [@tawachan](https://x.com/tawachan39) です。普段はWebフロントエンド、バックエンド、インフラを横断的に見ながら、チームの技術的な意思決定や開発効率化に取り組んでいます。

この記事では、**2025年の1年間で、PIVOTの小規模プロダクトチームがAI活用について何を試し、どう判断し、どんな課題に直面したのか**を振り返ります。

特定のAIツールの紹介や比較が主題ではありません。焦点を当てたいのは次の点です。

- 新しいツールが次々に出てくる中で、どう判断し
- どう調整し
- どうやってチームに馴染ませ、変えていったのか

という **意思決定と変化のプロセス**です。

---

## 1. はじめに

2025年は、AIツールが開発現場に急速に入ってきた1年でした。ただ、「AIを使ってみる」ことと「AIを前提にした開発体制を作る」ことの間には、想像以上に大きなギャップがありました。

PIVOTプロダクトチームも、まだ手探りの状態です。試しては調整し、変えてはまた別の方法を試す。その繰り返しの中で、少しずつ自分たちなりのやり方を探している段階です。

この記事は、その試行錯誤の過程を整理したものです。「こうすればうまくいく」という話ではなく、**今もまだ模索中のチームが、どんなことを試してきたか**という記録として読んでもらえればと思います。

---

## 2. 年初のスタート地点と初期の試行錯誤（1月〜春）

### 2.1 2025年1月時点のチームと課題意識

2025年1月、チームとして掲げていたテーマはシンプルでした。

> **AIを活用して、少しでもチームの生産性を上げられないか**

![2025年1月の生産性向上目標資料](/images/2025-01-productivity-goal.png)

*年初に共有した「開発生産性2倍」という目標。LLMの活用が一つの手段として挙げられていた*

当時の状況は次のようなものでした。

- GitHub Copilotは日常的に使っている
- Devinが話題になり始めていた
- CursorエディタやClineなどのAIコーディングツールが注目され始めた頃
- チームとしては、まだキャッチアップしきれていない

「すごそうだけど、どう使えばいいのかはよく分からない」という、少し距離のある状態だったと思います。

### 2.2 最初の挑戦とつまずき

最初の具体的なアクションとして、**Devinを1ヶ月間、チームで試験導入**しました。

結果から言うと、チームの開発スタイルには合いませんでした。

理由はいくつかあります。

- 不慣れで使いこなせなかった
- 細かい修正や調整が思った通りにいかない
- クレジット消費が多く、コスト感が掴みにくい
- コストを意識することで、依頼そのものを躊躇してしまう

「理論上は強いが、日常の開発には乗らない」という印象でした。

その後、ClineやCursorなども試しました。しかし、これらは専用エディタやVSCode拡張という形態で、特にiOS/AndroidのAppエンジニアには馴染みづらい面がありました。TypeScriptやGoを書く自分でも、まだ部分的なタスクしか任せられない感じで踏み切れませんでした（今思えば、使い方を十分に理解できていなかった部分も多分にあったと思います）。

何より、**コストをかけたらそれに見合う成果が出る**という期待感をまだ持てていませんでした。AIに任せられる範囲が限定的で、費用対効果が見えづらい状態だったのです。

## 3. CLIベースエージェントとの出会い

### 3.1 Claude Code / Codex CLI がもたらした転換点

流れが変わり始めたのは、**CLIベースのコーディングエージェント**が登場してからです。最初に触ったのはClaude Codeでした。

この「CLIで操作する」という汎用的なインターフェースが、想像以上に大きな意味を持ちました。

- エディタに依存しない
- Webエンジニアだけでなく、iOS / Androidエンジニアにも使いやすい
- Xcode / Android Studioの外側から自然に使える

さらに、**出てくるコードの質が高い**ことが決定打でした。当時はCursorやGitHub Copilotのエージェントモードと比べても、Claude Codeの方が明らかに良いという印象がありました。

![Claude Codeの評価が高かったやり取り](/images/claude-code-positive-chat.png)
![Claude CodeがCopilotより良いと感じたやり取り](/images/claude-code-vs-copilot.png)

その後、Codex CLIも選択肢に入りました。ChatGPTの有料プランに同梱されていたことで、**定額の中で使える＝コストを意識しすぎず試せる**という安心感があり、併用しやすくなりました。

これが、**チーム全体で試してみるきっかけ**になりました。

### 3.2 ゆるく試し続けるスタンス

この段階で明確な意思決定をしたわけではなく、**気になったものはまず触ってみる**というゆるいスタンスで試行錯誤を続けていました。

モデルが変われば、**Claudeが強い日もあればGPTが強い日もある**。翌日には逆転することも普通にあって、トークン数の都合で「こっちの方が良い」と感じる場面もありました。使い勝手の不満もありつつ、改善されて気にならなくなることもある。**日々変わりすぎて、常に模索している**という感覚が近いです。

この「揺れ」を受け止めながら動けた背景には、チーム側の前提が大きかったと思います。

## 4. 変化を受け入れられたチームの前提（7〜8月）

### 4.1 フットワークの軽さと試行回数

7〜8月頃になると、チームの動き方に特徴が出てきます。

- 誰かが見つけた情報がすぐ共有される
- 試すことに対する心理的ハードルが低い
- 合意形成が軽く、意思決定が速い
- 変えること自体に抵抗が少ない

この姿勢が、**日々変わる評価に合わせて動き続ける**ことを支えていたのだと思います。

![KiroとClaude Codeの試行が共有される様子](/images/ai-tool-experiment-share.png)
![NotionからGitHubへの流れを試す話](/images/claude-code-notion-github-flow.png)
![CodexとClaude Codeの使い分けが話題になる様子](/images/codex-vs-claude-preference.png)

### 4.2 AI以前からあった姿勢との接続

このフットワークの軽さは、AIが理由で突然生まれたものではありません。

- スクラムをやっていた頃から、やり方は頻繁に変えていた
- スクラムイベントの内容も固定していなかった
- 「守るべき型」がまだ存在していなかった

その姿勢が、**変化の激しいAI領域に対応する土台になっている**のかもしれません。

## 5. AI活用に向けたワークフローの試行錯誤

### 5.1 ワークフローの見直し

AI活用を進めるにつれ、従来の開発プロセスとのズレを感じるようになりました。AIが動きやすい形に整えないと、期待する効果が出ないという感覚です。

そのため、**「AIが迷わない前提づくり」**を意識して手当てしていきました。

- 勉強会でユースケースと失敗例を共有
- cloud.md / agents.md などのコンテキストファイルを整備
- 指示の出し方そのものをチームで学ぶ

そして試験的に、**スクラムを一旦やめてみる**という判断をしました。

- 計画より「まずやる」
- 固定プロセスが変化の足枷になる
- 全員が「部下（AI）を持つ」前提で動く

スクラムからの移行については、別の記事で詳しく書いています。  
https://zenn.dev/pivotmedia/articles/engineer-as-manager-scrum-to-kanban

> **🖼 Slackスクショ（プレースホルダー）**
> _スクラムをやめる／やり方を変える話をしているスレッド。ネガティブではなく前向きなトーンのもの_

### 5.2 同期／非同期作業と役割の変化

Codexなどのリモート実行系ツールにより、作業の切り分けも変わっていきました。

- その場で相談しながら進める作業
- 非同期でAIに任せる作業

という線引きが自然と議論になり、**「人が詰める部分」と「AIに委ねる部分」を分ける**意識が強くなりました。

プラットフォーム（Web / App）ごとの向き不向きも含め、まだ正解は見えていませんが、**小規模チームだからこそ試しながら調整できる**環境にあります。

## 6. 役割拡張とチームの広がり

### 6.1 非エンジニアへのAI活用の展開

Claude Codeの影響は、エンジニアに留まりませんでした。

- QAエンジニアがPlaywrightのテスト自動化コードを書く
- 非エンジニアでも複雑なことができるようになる

これは単なる効率化ではなく、**役割の境界が変わりつつある兆し**かもしれません。

> **🖼 Slackスクショ（プレースホルダー）**
> _QAエンジニアがClaude Codeで書いたテストコードを共有し、周囲が驚いているやり取り_

### 6.2 タスク管理とAI前提の運用

タスク管理もAI前提に変わっていきました。

- 過去Issueの検索と統合
- Slackのやり取りから課題を整理
- IDを探さずに適切な人へアサイン

この流れで、**GitHub Projectsへの移行**も試みています。

> **🖼 Slackスクショ（プレースホルダー）**
> _GitHub Projects移行を試しているスレッド。「AIから操作しやすい」「これは相性いいかも」という反応_

## 7. 振り返りとこれから

### 7.1 1年を通しての変化と現状

1年前とは違う開発スタイルを模索していますが、まだ確立したとは言えません。振り返ると、**変化を受け入れる姿勢**を持ち続けようとしていることが、様々な試行錯誤につながっているのだと思います。ただし、これが正解なのかはまだ分かりません。

### 7.2 2026年に向けて考えていること

やりたいけどまだできていないことが山積みです。

- Git worktreeなど並行作業環境の最適化
- Claude Code Skillsなど特定ツールへの集中投資

一方で、性急に固定せず、引き続き試行錯誤を続けることも必要だと感じています。
