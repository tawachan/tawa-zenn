こんにちは。PIVOTのテックリード [@tawachan](https://x.com/tawachan39) です。普段はWebフロントエンド、バックエンド、インフラを横断的に見ながら、チームの技術的な意思決定や開発効率化に取り組んでいます。

この記事では、**2025年の1年間で、PIVOTの小規模プロダクトチームがAI活用について何を試し、どんな課題に直面し、どう変化していったのか**を振り返ります。

特定のAIツールの紹介や比較が主題ではありません。書きたいのは、**正解が見えない中で、チームが日々どんなことを試行錯誤してきたか**という記録です。

その試行錯誤で大事にしてきたのは、最新技術・トレンドへのキャッチアップ自体ではなく、**チームの課題に即して必要なもの、取り入れられそうなものは何かを考える姿勢**です。変革は恐れずとも地に足をつけて判断する、そのバランスを意識しながら進めてきました。


## 1. はじめに

2025年は、AIツールが開発現場に急速に入ってきた1年でした。ただ、「AIを使ってみる」ことと「AIを前提にした開発体制を作る」ことの間には、想像以上に大きなギャップがありました。

PIVOTプロダクトチームも、まだ手探りの状態です。試しては調整し、変えてはまた別の方法を試す。その繰り返しの中で、少しずつ自分たちなりのやり方を探している段階です。

この記事は、その試行錯誤の過程を整理したものです。「こうすればうまくいく」という話ではなく、**今もまだ模索中のチームが、どんなことを試してきたか**という記録として読んでもらえればと思います。


## 2. 年初のスタート地点と初期の試行錯誤（1月〜春）

### 2.1 2025年1月時点のチームと課題意識

2025年1月、チームとして掲げていたテーマはシンプルでした。

> **AIを活用して、少しでもチームの生産性を上げられないか**

![2025年1月の生産性向上目標資料](/images/2025-01-productivity-goal.png)

*年初に共有した「開発生産性2倍」という目標。LLMの活用が一つの手段として挙げられていた*

当時の状況は次のようなものでした。

- GitHub Copilotは日常的に使っている
- Devinが話題になり始めていた
- CursorエディタやClineなどのAIコーディングツールが注目され始めた頃
- チームとしては、まだキャッチアップしきれていない

「すごそうだけど、どう使えばいいのかはよく分からない」という、少し距離のある状態だったと思います。

### 2.2 最初の挑戦とつまずき

最初の具体的なアクションとして、**Devinを1ヶ月間、チームで試験導入**しました。

結果から言うと、チームの開発スタイルには合いませんでした。

理由はいくつかあります。

- 不慣れで使いこなせなかった
- 細かい修正や調整が思った通りにいかない
- クレジット消費が多く、コスト感が掴みにくい
- コストを意識することで、依頼そのものを躊躇してしまう

「理論上は強いが、日常の開発には乗らない」という印象でした。

その後、ClineやCursorなども試しました。しかし、これらは専用エディタやVSCode拡張という形態で、特にiOS/AndroidのAppエンジニアには馴染みづらい面がありました。TypeScriptやGoを書く自分でも、まだ部分的なタスクしか任せられない感じで踏み切れませんでした（今思えば、使い方を十分に理解できていなかった部分も多分にあったと思います）。

何より、**コストをかけたらそれに見合う成果が出る**という期待感をまだ持てていませんでした。AIに任せられる範囲が限定的で、費用対効果が見えづらい状態だったのです。


## 3. CLIベースAIエージェントとの出会い

### 3.1 Claude Code / Codex CLI がもたらした転換点

流れが変わり始めたのは、**CLIベースのコーディングエージェント**が登場してからです。最初に触ったのはClaude Codeでした。

この「CLIで操作する」という汎用的なインターフェースが、想像以上に大きな意味を持ちました。

- エディタに依存しない
- Webエンジニアだけでなく、iOS / Androidエンジニアにも使いやすい
- Xcode / Android Studioの外側から自然に使える

さらに、**出てくるコードの質が高い**ことが決定打でした。当時はCursorやGitHub Copilotのエージェントモードと比べても、Claude Codeの方が明らかによいという印象がありました。

![Claude Codeの評価が高かったやり取り](/images/claude-code-positive-chat.png)
![Claude CodeがCopilotよりよいと感じたやり取り](/images/claude-code-vs-copilot.png)

その後、Codex CLIも選択肢に入りました。ChatGPTの有料プランに同梱されていたことで、**定額の中で使える＝コストを意識しすぎず試せる**という安心感があり、併用しやすくなりました。

これが、**チーム全体で試してみるきっかけ**になりました。

### 3.2 ゆるく試し続けるスタンス

この段階で明確な意思決定をしたわけではなく、**気になったものはまず触ってみる**というゆるいスタンスで試行錯誤を続けていました。

モデルが変われば、**Claudeが強い日もあればGPTが強い日もある**。翌日には逆転することも普通にあって、日々評価が揺れる。加えて、**向き不向きの感覚**もありました。たとえば **トークン数ではCodex CLIに強みがある**し、**柔軟さやスピードで直すならClaude Code**という肌感もある。使い勝手の不満もありつつ、改善されて気にならなくなることもある。**日々変わりすぎて、常に模索している**という感覚が近いです。

この「揺れ」を受け止めながら動けた背景には、チーム側の前提が大きかったと思います。


## 4. 変化を受け入れられたチームの前提（7〜8月）

### 4.1 フットワークの軽さと試行回数

7〜8月頃になると、チームの動き方に特徴が出てきます。

- 誰かが見つけた情報がすぐ共有される
- 試すことに対する心理的ハードルが低い
- 合意形成が軽く、意思決定が速い
- 変えること自体に抵抗が少ない

この姿勢が、**日々変わる評価に合わせて動き続ける**ことを支えていたのだと思います。

![KiroとClaude Codeの試行が共有される様子](/images/ai-tool-experiment-share.png)
![NotionからGitHubへの流れを試す話](/images/claude-code-notion-github-flow.png)
![CodexとClaude Codeの使い分けが話題になる様子](/images/codex-vs-claude-preference.png)

### 4.2 AI以前からあった姿勢との接続

このフットワークの軽さは、AIが理由で突然生まれたものではありません。振り返ってみると、もともとのチームの動き方が結果的に功を奏した面もあったと思います。

- スクラムをやっていた頃から、やり方は頻繁に変えていた
- スクラムイベントもやり方や時間などを日々模索し、柔軟に変えていた
- 「守るべき型」がまだ存在していなかった

その姿勢が、**変化の激しいAI領域に対応する土台になっている**のかもしれません。


## 5. AIに合わせたチームの変化

### 5.1 ワークフローとプロセスの見直し

AI活用を進めるにつれ、従来の開発プロセスとのズレを感じるようになりました。AIが動きやすい形に整えないと、期待する効果が出ないという感覚です。

そのため、「**AIが迷わない前提づくり**」を意識して手当てしていきました。

- 社内勉強会でユースケースと失敗例を共有
- `CLAUDE.md` / `AGENTS.md` などのコンテキストファイルを整備
- 指示の出し方そのものをチームで学ぶ

そして試験的に、**スクラムを一旦やめてみる**という判断をしました。

- 計画より「まずやる」
- 固定プロセスが変化の足枷になる
- 全員が「部下（AI）を持つ」前提で動く

スクラムからの移行については、別の記事で詳しく書いています。

https://zenn.dev/pivotmedia/articles/engineer-as-manager-scrum-to-kanban


### 5.2 作業の同期・非同期と役割の変化

Codexのような**バックグラウンドで動かせるコーディングエージェント**の登場で、作業の切り分けと流れが見え始めました。非同期自体は以前からできていましたが、**どこをAIに任せ、どこを人が詰めるか**の棲み分けが具体化した感覚があります。

例えば、こんな動き方が自然に増えていきました。

- SlackのスレッドからそのままCodexを呼び出す
- PM含め「気になったことは一旦AIに投げる」文化ができる
- 叩きはAIに作らせ、PRに仕上げるところはエンジニアが引き継ぐ
- 出社中や移動中に、細かい指示だけ先に投げておく
- その後、Codex CLIやClaude Codeで同期的に詰めていく

プラットフォーム（Web / App）ごとの向き不向きも含め、まだ正解は見えていませんが、**使いこなし方がチームの動き方と密接に関わる**ことは強く感じました。


### 5.3 役割拡張と非エンジニアへの展開

AIエージェントの影響は、実装担当に限らず、QAなど周辺の役割にも広がりました。

- QAエンジニアがPlaywrightのテスト自動化コードを書く
- 職種をまたいでAIとCLIツールを組み合わせ、柔軟に手を動かせるようになる

これは単なる効率化ではなく、**役割の境界が変わりつつある兆し**かもしれません。

![QAエンジニアがClaude Codeでテストコードを書いたときの反応](/images/qa-claude-code-test.png)

### 5.4 タスク管理とツール選択

ワークフローや役割の変化に伴い、タスク管理もAI前提に変わっていきました。Slackのやり取りから課題を拾い、過去Issueをつなげて整理する、といった流れが自然に増えています。**AIにとって解釈しやすい情報の置き場所と流れ**を意識し始めたことが、この時期の大きな変化でした。

その延長として、**NotionからGitHub Projectsへ移行する試み**も進めました。詳細は別の記事にまとめています。  
https://zenn.dev/pivotmedia/articles/pivot-notion-to-github-projects


## 6. 振り返りとこれから

### 6.1 1年を通しての変化と現状

間違いなく言えるのは、**年初の開発スタイルとは全く別物になっている**ということです。当時は想像もしていなかったことが、いまは日常になっています。キャッチアップに疲弊しているというより、**チームで楽しみながら変化に乗れている**感覚が強いです。そして明らかに、年初よりよくなっています。

### 6.2 2026年に向けて考えていること

まだまだよくできる余地はあるので、2026年も**楽しみながら試行錯誤する年**にしたいです。少人数でも生産性を落とさず、むしろ上げていくために、AIアシストをどこまで活かせるかを引き続き試していきます。

たとえば Git worktree を使った並列作業や tmux での同時実行など、**複数のAIエージェントを組み合わせる動き方**も知見として出てきています。チーム内でも少し試してはいますが、日常の仕事として馴染むところまではまだ届いていません。伸びしろは大きいと感じています。

同時に、Howだけに寄り過ぎず、**What**（どんなプロダクトにしていくか）にリソースを割ける状態を目指しています。AIの力で「やり方」を軽くし、**価値づくりに使える時間を増やす**方向で、どうなるかはまだわからないので手を動かしながら探っていきます。


## おわりに

この記事で書いたのは、まだ道半ばのチームの試行錯誤です。荒削りでもよくしていくこと自体が正義だと思っていて、**自分たちの課題に合う技術を、意味のあるソリューションとして取り入れていく**姿勢を大事にしています。最先端を追うこと自体が目的ではなく、**ちゃんと馴染む技術を積極的に選ぶ**。そして、技術に振り回されすぎないことも意識しながら、PIVOTプロダクトチームが今も前向きに試し続けている過程を記録しました。

同じように模索している他のチームと情報交換したり、自分たちの取り組みを振り返る材料になれば幸いです。
